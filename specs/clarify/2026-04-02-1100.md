**日期**: 2026-04-02

## 澄清決策摘要

| # | 問題 | 決策 | 說明 |
|---|------|------|------|
| Q1 | 管理員是否受線性觀看限制 | **A — 不受限** | 管理員可自由預覽所有章節，與現有管理員繞過邏輯一致 |
| Q2 | 章節完成定義 | **A — 維持手動按鈕** | 不改變現有完成機制，自動完成另案處理 |
| Q3 | 取消完成處理 | **C — 線性觀看下禁止取消完成** | 只允許前進不允許後退 |
| Q4 | 已完成但被鎖定章節顯示 | **A — 鎖定圖示優先** | 連續鏈斷裂時，後續章節統一顯示鎖定 |
| Q5 | 直接 URL 存取被鎖定章節 | **A — 導向未完成章節** | 自動跳轉到攤平排序中第一個未完成章節 + toast 提示 |
| Q6 | 完成後 UI 更新方式 | **B — 擴展 finishChapter.ts** | API 回傳 next_unlocked_chapter_id，前端 jQuery 更新鎖定圖示 |
| Q7 | toggle-finish API 驗證 | **A — API 加驗證** | 鎖定章節拒絕標記完成（403），線性觀看下拒絕取消完成（403） |

## 核心邏輯

### 解鎖演算法（連續完成鏈）
1. 取得課程所有章節的攤平排序（`ChapterUtils::get_flatten_post_ids()`）
2. 第一個章節永遠解鎖
3. 從第一個章節開始，逐一檢查是否已完成
4. 遇到第一個未完成的章節時停止 — 該章節為「下一個可觀看章節」
5. 該章節之後的所有章節均為鎖定狀態

### 資料儲存
- `enable_linear_viewing`: 課程 postmeta（`pc_yes_no` 型別，預設 `no`）
- 不需要新增資料表，完全依賴既有的 `pc_avl_chaptermeta` 中的 `finished_at`

### 影響的 API 端點
- `GET /progress/{course_id}`: 新增 `linear_viewing`、`unlocked_chapters`、`next_chapter_id` 回應欄位
- `POST /progress/toggle/{chapter_id}`: 新增線性觀看驗證（403 chapterLocked / unfinishForbidden） + 回傳 `next_unlocked_chapter_id`
- `PUT /courses/{id}`: 接受 `enable_linear_viewing` 參數
- `POST /courses`: 新建課程時 `enable_linear_viewing` 預設為 `no`

### 影響的模板/前端
- `single-pc_chapter.php`: 新增線性觀看存取控制，鎖定時導向
- `classroom/chapters.php`: 側邊欄章節列表渲染鎖定圖示
- `finishChapter.ts`: 擴展回應處理，更新下一章的鎖定圖示
