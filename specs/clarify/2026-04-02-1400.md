# 需求澄清紀錄 — 課程線性觀看功能

**Issue**: #165
**日期**: 2026-04-02
**參與者**: j7-dev, clarifier agent

---

## 澄清問題與決議

### Q1. 學員「取消完成」章節時的行為
- **決議**: **C — 禁止取消完成**
- **說明**: 開啟線性觀看的課程，不允許學員取消已完成的章節。toggle-finish API 在線性模式下只允許「標記完成」，不允許「取消完成」。
- **影響**: 需修改 `post_toggle_finish_chapters_with_id_callback` 加入線性模式判斷。前端已完成的章節不顯示「標示為未完成」按鈕。

### Q2. 父章節與內容章節的區分
- **決議**: **所有章節都參與線性順序**，即使是父章節也沒有內容也要計入
- **說明**: 不區分父章節/葉子章節，所有 `pc_chapter` post 按照扁平化的 `menu_order` 排序參與線性判定。
- **影響**: 使用 `ChapterUtils::get_flatten_post_ids()` 取得完整順序，無需過濾父章節。

### Q3. 管理員/講師是否繞過鎖定
- **決議**: **A — 管理員與講師完全繞過**
- **說明**: 具有 `manage_woocommerce` capability 或課程作者身份的用戶，不受線性鎖定限制。
- **影響**: 模板層和 API 層的攔截邏輯需加入權限判斷。

### Q4. 完成章節後的導航行為
- **決議**: **A — 顯示提示，不自動跳轉**
- **說明**: 彈出「已完成！下一章節已解鎖」的提示，讓學員自行決定是否前往下一章。
- **影響**: 修改 toggle-finish API 回傳 `next_chapter_id` 和 `next_chapter_unlocked`，前端顯示提示對話框。

### Q5. 設定儲存位置
- **決議**: **A — WC Product Meta**，meta_key 為 `enable_linear_mode`
- **說明**: 值為 `'yes'` / `'no'`，預設 `'no'`（關閉）。與現有的 `_is_course` 等 meta 一致。
- **影響**: 使用 `$product->get_meta('enable_linear_mode')` 讀取，`update_post_meta($product_id, 'enable_linear_mode', 'yes')` 寫入。

### Q6. 鎖定的伺服器端攔截方式
- **決議**: **B — 模板層 + REST API 雙重攔截**
- **說明**: 
  - 模板層：在 `single-pc_chapter.php` 渲染前檢查，鎖定時替換為提示頁面
  - API 層：`toggle-finish` 和章節內容 API 加入鎖定檢查
- **影響**: 需在模板和 API 兩處新增解鎖狀態判斷。

### Q7. 免費預覽章節豁免機制
- **決議**: **A — 第一期不做豁免機制**
- **說明**: 線性觀看為全有全無，所有章節都參與線性順序。
- **影響**: 無需額外豁免邏輯。

## 用戶補充需求

1. **JS 局部解鎖**: 用戶點完完成後，使用 JS 局部更新側邊欄的鎖定狀態，讓下一章可以存取，避免重新載入頁面。
2. **鎖定提示**: 上鎖的章節要有明確的提示給用戶，告知原因和下一步。
