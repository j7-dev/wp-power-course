# 澄清紀錄：Issue #173 課程線性觀看功能

**日期**: 2026-04-03
**來源**: GitHub Issue #173

## 澄清問題與回答

### Q1: 父章節（分組標題）是否算作「可完成」的章節？
**選項**: A) 跳過父章節，只看葉節點 / B) 父章節也要完成 / C) 視父章節是否有內容而定
**回答**: **攤平所有章節** — 不論是否為父章節，全部攤平(flat)計入線性解鎖順序

### Q2: 鎖定是否需要伺服器端強制執行？
**選項**: A) 前端 + 後端雙重鎖定 / B) 僅前端鎖定
**回答**: **A** — 前端 + 後端雙重鎖定，防止 URL 直接存取繞過

### Q3: 學員取消已完成章節時的行為？
**選項**: A) 顯示確認提示 / B) 不需要確認 / C) 僅影響後續章節時顯示確認
**回答**: **禁止取消** — 若啟用線性觀看功能，已完成的章節不能轉為取消（toggle 變單向）

### Q4: 課程中途開啟線性觀看，已有跳躍式進度的舊學員如何處理？
**選項**: A) 嚴格線性規則 / B) 已完成章節豁免 / C) 完全豁免舊學員
**回答**: **A** — 嚴格線性規則，已完成紀錄保留，但仍受前一章必須完成的限制

### Q5: 「前往下一章節」按鈕在下一章被鎖定時的行為？
**選項**: A) 按鈕禁用灰色 / B) 可點擊顯示提示 / C) 隱藏按鈕 / D) 完成後自動啟用
**回答**: **D** — 禁用狀態 + 完成後即時啟用

### Q6: 課程銷售頁是否需要標示「本課程為循序學習」？
**選項**: A) 不需要標示 / B) 顯示簡短標示 / C) 後續再做
**回答**: **A** — 不需要標示

### Q7: 管理員預覽模式是否受線性觀看限制？
**選項**: A) 管理員不受限但看得到鎖定 UI / B) 管理員也受限 / C) 管理員不顯示鎖定 UI
**回答**: **A** — 管理員完全不受限，但 UI 上仍顯示鎖頭圖示

## 確認的規格摘要

### 核心功能
- 每門課程可獨立開啟/關閉「線性觀看」模式（預設關閉）
- 開啟後學員必須按章節排列順序逐步完成
- 第一個章節預設為解鎖狀態

### 章節排序規則
- 所有章節攤平(flat)為線性序列，不區分父子章節
- 排序依據 `get_flatten_post_ids()` 的既有邏輯（menu_order ASC → ID DESC → date DESC）

### 鎖定機制
- **前端**：鎖頭圖示 + 點擊顯示友善提示
- **後端**：`single-pc_chapter.php` 模板渲染前檢查，鎖定時導向最新可存取章節
- 管理員（`manage_woocommerce`）不受後端鎖定限制，但前端顯示鎖定 UI

### 完成行為變更
- 啟用線性觀看時，「標示為未完成」按鈕**隱藏/禁用**（單向操作，只能標記完成）
- 關閉線性觀看時，恢復雙向 toggle（與現有行為一致）

### 「前往下一章節」按鈕
- 線性觀看模式下，未完成當前章節時按鈕禁用（灰色）
- 完成當前章節後，按鈕透過 JS 即時啟用（不需重新載入）

### 模式切換
- 關閉線性觀看：所有章節立即恢復自由觀看，已記錄的完成進度不受影響
- 中途開啟線性觀看：嚴格線性規則生效，即使之前已完成跳躍式章節

### 資料模型
- 新增課程 postmeta: `linear_chapter_mode`（`yes` / `no`，預設 `no`）
- 不需要新增資料表，利用既有的 `pc_avl_chaptermeta.finished_at` 判斷解鎖狀態

- 課程銷售頁標示（不需要）
