# Clarify Session 2026-04-20 17:48

## Idea

Issue #206：**課程影片觀看完後無法重複觀看**

學員已完成一次觀看後，由於續播功能將 `last_position_seconds` 保留在影片極接近片尾的位置（例如 580/600 秒，超過 `FINISH_THRESHOLD=0.95`），下次從「重看」CTA 重新進入章節 Player 時：

1. VidStack Player 被 seek 到接近片尾的秒數（~580）
2. 影片很快播放到 ended 事件
3. `<Ended />` 倒數遮罩（`absolute inset-0 w-full h-full bg-black/50`）蓋住整個 Player
4. VidStack 的進度條、控制列、播放按鈕全部被遮蔽，學員**無法拖曳進度條回到開頭、無法暫停、無法重播**
5. 5 秒倒數結束後，`window.location.href = next_post_url` 強制跳到下一章節

根因：續播秒數 ≥ FINISH_THRESHOLD × duration 時，Ended 遮罩會在極短時間內觸發，且遮罩沒有「取消自動跳轉」或「重看本章」的出口，導致學員被綁架到下一章，無法達成「重看」的意圖。

## 背景檔案

- `js/src/App2/Ended.tsx` — 倒數遮罩元件，使用 `setInterval` 遞減 5 秒，結束時 `window.location.href = next_post_url`
- `js/src/App2/Player.tsx` — 管理 `isEnded` state、`onEnded` 事件、seek 邏輯
- `js/src/App2/hooks/useChapterProgress.ts` — 續播秒數 GET / POST
- 關聯規格：
  - `specs/features/progress/續播至上次觀看秒數.feature`
  - `specs/features/progress/影片進度自動完成章節.feature`
  - `specs/features/course/繼續觀看課程CTA.feature`
  - `specs/open-issue/004-resume-playback-position.md`

## Q&A

### Q1 取消遮罩的 UX 主軸

- 題目：Ended 倒數遮罩應該給學員什麼樣的出口？
- 選項：
  - A：只提供「取消自動跳轉」一個按鈕
  - B：只提供「重看本章」一個按鈕
  - C：**同時提供「取消自動跳轉」＋「重看本章」兩個按鈕**
  - D：其他
- **最終決策：C**
- 理由：兩種意圖都是學員可能想做的（停留 vs 重看），兩顆按鈕成本低且語義清楚；「取消」是被動出口，「重看」是主動出口，一次滿足兩種期望。

### Q2 遮罩範圍策略

- 題目：遮罩要不要把 VidStack 進度條 / 控制列一併蓋住？
- 選項：
  - A：遮罩只蓋中央倒數文字區（進度條仍可操作）
  - B：遮罩蓋全屏但控制列透過 `pointer-events: auto` 穿透
  - C：**遮罩全屏，必須先按「取消自動跳轉」解鎖才能操作進度條**
  - D：其他
- **最終決策：C**
- 理由：視覺乾淨，學員意圖明確後（按「取消」）才釋出控制權，避免誤觸倒數被提前中止；搭配 Q5 的 `onSeeking` 事件作為隱性備援，同時滿足「明確操作」與「拖動自動解鎖」。

### Q3 「取消自動跳轉」後的播放狀態

- 題目：按下「取消自動跳轉」之後，影片要停在哪裡？
- 選項：
  - A：**遮罩消失，影片停在片尾（paused at ended）**
  - B：遮罩消失，影片自動 seek 回 0 並 pause
  - C：遮罩消失，影片自動 seek 回 0 並 play
  - D：其他
- **最終決策：A**
- 理由：「取消跳轉」語義就是「我不想走」，不是「我要重看」。停在片尾讓學員自由選擇下一步（拖進度條、按重看、關頁面）。若要重看則改按「重看本章」按鈕。

### Q4 續播到片尾仍 5 秒自動跳下一章

- 題目：續播續到片尾後要不要保留 5 秒自動跳轉？
- 選項：
  - A：**照舊 5 秒倒數後自動跳下一章（首次觀看與續播行為一致）**
  - B：續播情境不自動跳，只顯示兩顆按鈕，等學員選
  - C：續播且已完成過的章節才不自動跳
  - D：其他
- **最終決策：A**
- 理由：保持首次觀看與續播行為的一致性，降低認知負擔。出口（取消 / 重看 / seek）已足夠；自動跳轉是常見學習流的省事設計。

### Q5 倒數期間拖拉進度條自動中止倒數

- 題目：倒數中學員拖動進度條，要不要自動中止倒數？
- 選項：
  - A：**是，偵測 `onSeeking` 事件即中止倒數**
  - B：不中止，等 5 秒到或按鈕被點
  - C：中止倒數但不隱藏遮罩
  - D：其他
- **最終決策：A**
- 理由：拖進度條已是「我不想跳轉」的強烈意圖訊號；讓 `onSeeking` 作為 Q2=C 全屏遮罩的隱性備援，兩顆按鈕仍是明確出口。

### Q6 `last_position_seconds` 極接近 duration 的處理

- 題目：續播秒數接近或超過 duration 時要不要特別處理？
- 選項：
  - A：**照原秒數 seek，ended 後顯示可取消遮罩**（沿用 Q1-Q5 設計）
  - B：seek 前先夾擠到 duration − 5 秒
  - C：seek 到 0 並從頭播放
  - D：其他
- **最終決策：A**
- 理由：`004-resume-playback-position.md` Example「已完成且秒數接近片尾仍正常續播」已定義為續播邏輯；用 Ended 遮罩的出口處理「太快結束」的問題，規格一致性優於特殊處理。

### Q7 測試策略

- 題目：這次修復要補什麼等級的測試？
- 選項：
  - A：**Playwright E2E 新增 3 條路徑（取消按鈕 / 拖拉進度條 / 最後一章無遮罩）**
  - B：只補 1 條 E2E（重看後按取消可停留）
  - C：React Integration Test 取代 E2E
  - D：其他
- **最終決策：A**
- 理由：與 `004-resume-playback-position.md` Q15 的 E2E 核心路徑策略一致；三條路徑涵蓋按鈕、隱性備援、最後一章 edge case，對 UX 回歸保護足夠。

### Q8 是否 patch `影片進度自動完成章節.feature`

- 題目：這次修復要更新哪些 feature 檔？
- 選項：
  - A：只更新 `續播至上次觀看秒數.feature`
  - B：只更新 `影片進度自動完成章節.feature`
  - C：**兩個 feature 都要更新**
  - D：其他
- **最終決策：C**

  - 自動完成 feature：既有「自動完成 API fire-and-forget」Rule 下的 Ended 倒數行為描述需要修正，**不再強制**跳下一章，必須提供出口

### Q9 順便修 `Ended.tsx` setInterval 競態條件

- 題目：既有 `Ended.tsx` 的 `setInterval` 有潛在競態（清除 interval 後仍可能執行 `window.location.href`），要不要順便修？
- 選項：
  - A：**順便修（加 `isCancelledRef` 守衛 `window.location.href`）**
  - B：不修，保留既有行為
  - C：改用 `setTimeout` 而非 `setInterval`
  - D：其他
- **最終決策：A**
- 理由：既然要改這個檔，把競態一起修成本極低且避免之後出 bug；加 `isCancelledRef` 是最小改動，向後相容。

### Q10 回歸保護

- 題目：首次觀看自然播放完的 5 秒自動跳轉是否要加 E2E 保護？
- 選項：
  - A：**要，必須加 Playwright 測試保護**
  - B：靠手動 QA 即可
  - C：只加 React IT，不加 E2E
  - D：其他
- **最終決策：A**
- 理由：本次修改會動到 Ended 遮罩的倒數核心邏輯（新增 `isCancelledRef`、`onSeeking` 中止機制），有回歸風險；加一條 E2E 保護首次觀看→自然跳下一章的主流程是必要成本。

## Q2 × Q5 整合說明

Q2=C（遮罩全屏，按鈕解鎖才能操作）與 Q5=A（拖進度條中止倒數）乍看衝突，實際整合後的完整行為：

- **視覺層**：遮罩 `absolute inset-0` 疊在 VidStack 控制列之上（`z-index` 足夠壓住 VidStack 的 controller）。學員看到的是「乾淨的遮罩 + 兩顆按鈕 + 倒數秒數」。
- **互動層**：
  - 學員點「取消自動跳轉」→ 遮罩消失，影片停在片尾（paused）
  - 學員點「重看本章」→ seek 到 0 + play，遮罩消失
  - 學員透過 VidStack 原生 hover/拖拉操作進度條 → `onSeeking` event 被 Player.tsx 捕獲 → 呼叫 `onCancel()` → 遮罩消失（等同按下「取消自動跳轉」）
  - 5 秒倒數結束且上述三者皆未觸發 → `window.location.href = next_post_url`
- **關鍵**：`onSeeking` 是隱性備援，不是主要 UX。學員看到的 UX 仍然是「全屏遮罩 + 兩顆按鈕」。只是如果學員直覺地去拖 VidStack 原生控制列（即便被遮罩覆蓋，VidStack 可能仍透過鍵盤或 hover 區域接收到事件），遮罩會優雅讓出。

**日期**：2026-04-20

**影響決策**：Q1 由 C → B；Q2 / Q3 / Q5 連動廢止
**仍維持**：Q4 / Q6 / Q7（條數調整）/ Q8 / Q9 / Q10

### BUG 觀察

人工測試「取消自動跳轉」流程時發現兩個難解的 VidStack 行為：

- **2-1 播放按鈕消失**：按下「取消自動跳轉」後，遮罩雖然消失，但 VidStack 在 `ended` 狀態下不會顯示原生控制列的播放按鈕，用戶被鎖在 paused at ended，除了關頁面無其他出口。
- **2-2 拖拉進度條觸發 progress API 把進度條 seek 回片尾**：即使 2-1 被繞過，用戶拖拉進度條時，`useChapterProgress` 的續播 GET 會重新取得 `last_position_seconds`（接近片尾），並立刻 seek 回片尾，形成「拖一下就彈回」的死循環。

### 新決策

- **Q1：C → B** — 只保留「重看本章」作為唯一明確出口。因為 `currentTime = 0 + play()` 立刻脫離 ended 狀態，天然繞過 2-1 與 2-2。
- **Q2（遮罩範圍策略）廢止** — 沒有「取消」按鈕需要「解鎖」，遮罩範圍策略失去意義。
- **Q3（取消後播放狀態）廢止** — 按鈕已移除。
- **Q5（onSeeking 隱性備援）廢止** — 若保留 `onSeeking` guard，在遮罩消失但媒體層仍 ended 的狀態下，用戶拖拉會觸發 progress API 把進度條 seek 回片尾，形成 BUG 2-2。故 `Player.tsx` 連同移除 `onSeeking` 監聽。

### Q7 條數調整

原 Q7=A 要求「3 條新增 + 1 條回歸」，post-test 後縮減為「1 條新增（重看本章）+ 1 條回歸（首次觀看）」，共 2 條。被刪除的 E2E：
- Test 1「取消自動跳轉」— Q1 連動廢止
- Test 3「倒數期間拖進度條自動中止倒數」— Q5 連動廢止
- Test 5「onSeeking 不誤觸倒數中止」— Q5 連動廢止
